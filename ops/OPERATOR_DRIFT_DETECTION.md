# OPERATOR_DRIFT_DETECTION.md
### Purpose  
Defines the mechanisms and procedures for detecting operator drift during all PasteShield operations.  
Ensures the operator remains attentive, aligned with doctrine, and free from complacency.

---------------------------------------------------------------------

## 0. Rationale

Operator drift is a silent threat.  
It emerges gradually when:

- workflows become routine  
- anomalies become rare  
- systems appear stable  
- vigilance decays without notice  

Drift must be detected early, surfaced clearly, and corrected immediately.

---------------------------------------------------------------------

## 1. Definition of Operator Drift

Operator drift is any deviation from:

- required attention level  
- required anomaly sensitivity  
- required classification accuracy  
- required doctrine adherence  
- required cross‑checking discipline  

Drift is not a failure of skill — it is a failure of vigilance.

---------------------------------------------------------------------

## 2. Drift Indicators

Drift may be detected through:

**Behavioral Indicators**  
- Reduced anomaly capture  
- Delayed anomaly classification  
- Over‑trust in system behavior  
- Skipped cross‑checks  
- Reduced note detail  
- Increased assumptions  

**Procedural Indicators**  
- Missing daily log details  
- Missing context fields  
- Incomplete anomaly entries  
- Inconsistent terminology  
- Retroactive edits  

**Cognitive Indicators**  
- “It’s probably fine” thinking  
- Expectation of correctness  
- Reduced skepticism  
- Reduced friction sensitivity  

---------------------------------------------------------------------

## 3. Drift Detection Mechanisms

PasteShield operations must include:

- Daily log review for completeness  
- Weekly anomaly pattern review  
- Cross‑checking doctrine against observed behavior  
- Monitoring for reduced anomaly frequency  
- Monitoring for reduced operator notes  
- Monitoring for skipped steps in workflows  

Any deviation triggers drift investigation.

---------------------------------------------------------------------

## 4. Drift Response Procedure

When drift is detected:

1. Stop normal work  
2. Identify drift source  
3. Review recent anomalies  
4. Review doctrine  
5. Review test matrix  
6. Re‑establish vigilance baselines  
7. Resume work only after alignment is restored  

Drift must never be ignored.

---------------------------------------------------------------------

## 5. Drift Prevention Strategies

- Maintain friction in workflows  
- Preserve visible imperfections  
- Keep failures big and obvious  
- Require explicit classification  
- Require explicit operator notes  
- Require explicit cross‑checks  
- Rotate tasks to avoid habituation  
- Periodically re‑read doctrine  

---------------------------------------------------------------------

## 6. Enforcement

This rule applies to:

- All operator workflows  
- All soak logs  
- All anomaly classifications  
- All doctrine updates  
- All test matrix updates  
- All release and rollback procedures  

Operator drift is a safety‑critical condition.
