# OPERATOR_RHYTHM_AND_FRICTION.md
### Purpose  
Defines the required rhythm and friction patterns that maintain operator engagement, prevent habituation, and preserve vigilance during all PasteShield operations.

---------------------------------------------------------------------

## 0. Rationale

Operators lose vigilance when:

- workflows become too smooth  
- systems behave too perfectly  
- friction disappears  
- cadence becomes monotonous  

A controlled level of friction and a deliberate operational rhythm prevent drift and maintain attention.

---------------------------------------------------------------------

## 1. Core Principle

**Operator attention is preserved through structured rhythm and intentional friction.**

The system must never allow the operator to fall into passive trust.

---------------------------------------------------------------------

## 2. Rhythm Requirements

Operator workflows must include:

- Clear start and stop boundaries  
- Predictable checkpoints  
- Regular anomaly classification  
- Daily context resets  
- Weekly consolidation cycles  
- Monthly summary cycles  

Rhythm creates structure.  
Structure creates awareness.

---------------------------------------------------------------------

## 3. Friction Requirements

Friction must be:

- Intentional  
- Bounded  
- Visible  
- Meaningful  
- Non‑catastrophic  

Friction must **never**:

- Obscure information  
- Create ambiguity  
- Introduce silent failure  
- Impede anomaly detection  
- Undermine doctrine  

---------------------------------------------------------------------

## 4. Acceptable Friction Patterns

Acceptable:

- Required anomaly classification  
- Required operator notes  
- Required cross‑checks  
- Required doctrine updates  
- Required test matrix updates  
- Required daily/weekly/monthly reviews  

Unacceptable:

- Hidden steps  
- Implicit assumptions  
- Auto‑corrections without visibility  
- Silent normalization  
- Invisible state changes  

---------------------------------------------------------------------

## 5. Operator Responsibilities

Operator must:

- Maintain cadence  
- Embrace friction as a safety feature  
- Avoid shortcuts  
- Avoid skipping steps  
- Avoid assumptions  
- Preserve explicitness  
- Re‑establish rhythm after interruptions  

---------------------------------------------------------------------

## 6. System Responsibilities

System must:

- Provide consistent cadence  
- Surface friction intentionally  
- Avoid over‑automation  
- Avoid smoothing away critical steps  
- Preserve operator involvement  
- Preserve operator decision‑making authority  

---------------------------------------------------------------------

## 7. Enforcement

This rule applies to:

- All ops/ workflows  
- All soak logs  
- All anomaly classifications  
- All doctrine updates  
- All test matrix updates  
- All release and rollback procedures  
- All monitoring procedures  

No component is exempt.
